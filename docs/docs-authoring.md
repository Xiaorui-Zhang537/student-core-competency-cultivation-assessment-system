# 文档写作规范

本文档用于约束本仓库 `docs/` 下 Markdown 的结构与写法，目标是：

- 让新同学能按“可跑通 -> 可理解 -> 可修改”的路径阅读
- 让文档长期与代码保持一致，减少“写完就过期”
- 让页面结构统一，便于搜索与导航

## 1. 范围与原则

- 本规范针对 `docs/` 下的项目文档；不维护 `docs/node_modules/**`、`docs/.vitepress/dist/**` 等依赖或生成物。
- 文档优先级：可操作性 > 完整性 > 漂亮措辞。
- 约束：“后端即真理”。接口/字段/枚举以后端实际实现与 Swagger 为准，文档用于解释“怎么用 + 为什么这样设计 + 常见坑”。

## 2. 页面结构约定（必须）

- 每个页面必须只有一个一级标题 `#`，且放在文件最顶部（frontmatter 之后）。
- 标题层级只用 `##`、`###`（避免跳级）。
- 页面建议包含以下固定块（按需要出现）：
  - 目标（Goal）
  - 前置条件（Prerequisites）
  - 步骤（Steps）
  - 验证（Verify）
  - 常见问题（Troubleshooting）
  - 相关链接（Related）

## 3. Frontmatter（推荐）

对外展示/索引页建议写 frontmatter（VitePress 支持）：

```md
---
title: 页面标题
description: 一句话说明这页解决什么问题
outline: [2, 3]
---
```

## 4. 代码块与示例

- 代码块必须标注语言：`bash`、`ts`、`json`、`yaml`、`sql`、`java`、`vue`、`mermaid`。
- 示例必须可复制运行（除非明确标注为伪代码）。
- 涉及环境变量/域名/端口时，优先给出本地默认值（例如 `http://localhost:8080/api`）。

## 5. 告警块（推荐）

用告警块表达“重要约束/风险/提示”，避免埋在段落里：

```md
:::tip 提示
这是一条实践建议。
:::

:::warning 注意
这里可能踩坑，说明原因和解决方式。
:::

:::danger 严禁
这里是强约束（安全/合规/数据口径）。
:::
```

## 6. 链接与路径

- 站内链接用 VitePress 绝对路径（不带 `.md`），例如：`/backend/api/index`。
- 代码路径用反引号包裹，且尽量精确到文件：`frontend/src/api/auth.api.ts`。
- 不在文档里粘贴超长的源码；优先解释“入口文件 + 关键函数 + 调用链”。

## 7. API 文档模板（后端）

每个控制器页建议按以下结构编写：

```md
# XXX API

> 以 Swagger 为准：`http://localhost:8080/api/swagger-ui.html`

## 1. 端点总览

## 2. 认证与权限

## 3. 端点详情

### 3.1 GET /api/...
- 权限：
- Query：
- Body：
- Response（示例）：
- 常见错误：
```

## 8. 前端 API 文档模板（`frontend/src/api/*.ts`）

每个文件对应一个页面：

```md
# `xxx.api.ts`

## 1. 职责与使用场景

## 2. 方法清单

## 3. 最小调用示例

## 4. 常见坑与约束
```

## 9. 数据契约/规格文档模板（业务口径）

用于“数据口径/契约/治理规则”的页面建议结构：

```md
# 主题：数据契约（vX）

## 1. 目标与非目标

## 2. 版本与兼容策略

## 3. 字段定义（表格）

## 4. 流程与边界条件

## 5. 示例（JSON）

## 6. 审计与治理规则
```

## 10. 更新策略（避免过期）

- 新增/修改接口：必须同步更新对应 `docs/backend/api/*.md` 与 `docs/frontend/api/*.md`。
- 新增/修改数据表/字段：必须同步更新 `docs/models/*.md` 与相关接口文档。
- 重要行为口径变更：必须同步更新契约文档，并升级 schemaVersion（如适用）。
